# Well Finder — backend pipeline

The map at `/well-finder` reads a single pre-tiled `wells.pmtiles` archive
from Firebase Storage. Two pieces produce that file:

| Component                | What it does                                                                                                                                                                                   | Where it lives                                    |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| `fetchRrcWells`          | Cloud Function (scheduled monthly, 1st of month 09:00 UTC). Pulls every well from RRC ArcGIS Layer 1 with pagination, normalizes to GeoJSON, gzips, uploads to `well-finder/wells.geojson.gz`. | `functions/src/wellFinder/fetchRrcWells.ts`       |
| `triggerPmtilesBuild`    | Storage trigger. Fires when `wells.geojson.gz` is (over)written. Invokes the Cloud Run tippecanoe service which produces `wells.pmtiles`.                                                      | `functions/src/wellFinder/triggerPmtilesBuild.ts` |
| `well-finder-tippecanoe` | Cloud Run service. Custom Docker image with tippecanoe (Felt fork). Reads GeoJSON from Storage, runs tippecanoe, writes PMTiles back.                                                          | `cloudrun-tippecanoe/`                            |

## One-time setup

### 1. Deploy the Cloud Run tippecanoe service

```bash
gcloud run deploy well-finder-tippecanoe \
  --source cloudrun-tippecanoe \
  --region us-central1 \
  --memory 2Gi --cpu 2 --timeout 600 \
  --no-allow-unauthenticated
```

First build takes ~5 min (compiles tippecanoe from source).

### 2. Wire IAM and the secret

```bash
PROJECT_ID="<your-project-id>"
RUN_SA="$PROJECT_ID@appspot.gserviceaccount.com"

# Allow the Functions runtime SA to invoke the Cloud Run service
gcloud run services add-iam-policy-binding well-finder-tippecanoe \
  --region us-central1 \
  --member="serviceAccount:$RUN_SA" \
  --role="roles/run.invoker"

# Store the Cloud Run URL as a Secret Manager secret
RUN_URL=$(gcloud run services describe well-finder-tippecanoe \
  --region us-central1 --format='value(status.url)')

echo -n "$RUN_URL" | gcloud secrets create WELL_FINDER_TIPPECANOE_URL --data-file=-

gcloud secrets add-iam-policy-binding WELL_FINDER_TIPPECANOE_URL \
  --member="serviceAccount:$RUN_SA" \
  --role="roles/secretmanager.secretAccessor"
```

### 3. Deploy the Cloud Functions

```bash
firebase deploy --only functions:fetchRrcWells,functions:triggerPmtilesBuild
```

### 4. Configure Storage for browser fetch

`wells.pmtiles` is read from the browser. Either:

**A. Make it public** (simplest for an admin-only tool):

```bash
gcloud storage buckets add-iam-policy-binding gs://<your-default-bucket> \
  --member=allUsers \
  --role=roles/storage.legacyObjectReader
```

**B. Or use a signed URL** generated by a Cloud Function and refreshed periodically.

Then set the file's CORS so MapLibre can do range requests:

```json
[
  {
    "origin": ["*"],
    "method": ["GET", "HEAD"],
    "responseHeader": ["Content-Type", "Content-Range", "Range", "Accept-Ranges"],
    "maxAgeSeconds": 3600
  }
]
```

```bash
gcloud storage buckets update gs://<your-default-bucket> --cors-file=cors.json
```

### 5. Set the frontend env var

In `.env` or `.env.production` at the repo root:

```
VITE_WELL_FINDER_PMTILES_URL=https://storage.googleapis.com/<your-bucket>/well-finder/wells.pmtiles
```

The frontend will switch from live RRC queries to PMTiles.

## Running it

### Trigger the first snapshot manually

The scheduled function runs the 1st of each month. To prime it now:

```bash
gcloud functions call fetchRrcWells --region us-central1
```

That writes `wells.geojson.gz` → triggers `triggerPmtilesBuild` → Cloud Run
runs tippecanoe → uploads `wells.pmtiles`.

### Verify the file exists

```bash
gsutil ls gs://<your-default-bucket>/well-finder/
```

Should show:

- `wells.geojson.gz` (latest snapshot, ~10–20 MB)
- `snapshots/wells-YYYY-MM.geojson.gz` (this month's history copy)
- `wells.pmtiles` (the tile archive, ~80 MB)

## Local dev without the pipeline

If the backend isn't deployed (or `VITE_WELL_FINDER_PMTILES_URL` is unset),
the frontend falls back to **live RRC ArcGIS queries**, paginated up to 50K
records for the visible status filters. You'll see a small "● Live RRC"
badge in the top-left status banner. This is what Phase 0 ran on.

## Cost

| Component                            | Cadence                                        | Estimated $/mo |
| ------------------------------------ | ---------------------------------------------- | -------------: |
| `fetchRrcWells` Cloud Function       | Monthly, ~10 min @ 1 GiB                       |         <$0.05 |
| `triggerPmtilesBuild` Cloud Function | Monthly, <1 min @ 256 MiB                      |         <$0.01 |
| `well-finder-tippecanoe` Cloud Run   | Monthly, ~60 sec @ 2 vCPU + 2 GiB              |         <$0.10 |
| Firebase Storage                     | ~100 MB stored + ~5 GB egress (admin browsing) |         <$0.50 |
| **Total**                            |                                                |     **<$1/mo** |
